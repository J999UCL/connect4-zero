Game model

Gravity actions on a 4-layer cube

A move chooses an (x, y) column. Gravity places the piece at the next open z height, where z=0 is the bottom.

Action mapping

The policy space is intentionally small: 16 column actions, not 64 cell placements. This matches the physical 4x4x4 Connect Four rule variant and keeps MCTS and later policy heads simple. 

action = 4 * x + y
x = action // 4
y = action % 4

Action 5 selects column x=1, y=1.

Phase 1

The vectorized game engine

Connect4x4x4Batch is the rule engine. It is not a player or a training loop. It owns batched state tensors and knows how to apply one move per game.

board

torch.int8, shape (B, 4, 4, 4)
B games x axis y axis z axis

Values are 1, -1, or 0. The board is always from the next player-to-move perspective.

heights

torch.int8, shape (B, 4, 4)
0103 2410 0031 4210

Each number is the next open z. A value of 4 means the column is full and illegal.

done and outcome

done: bool (B,), outcome: int8 (B,)
live win draw live

outcome=1 means the player who just moved won. outcome=0 means unfinished or draw.

What step(actions) does

  1. Validate one action per game in the batch.
  2. Convert each action to x, y.
  3. Use heights[b, x, y] to find the landing z.
  4. Place a +1 stone for legal active games.
  5. Check wins immediately, before perspective flips.
  6. Check draws from full boards with no winning line.
  7. Flip non-terminal boards with board *= -1.

legal_mask()

Returns a (B, 16) boolean mask. Full columns and completed games are false.

check_wins()

Gathers the 76 static lines, sums each length-4 line, and detects sums of 4 or -4.

is_draw()

Requires every column to be full and neither player to have a winning line.

clone()

Makes a deep state copy, which is what search uses when expanding hypothetical moves.

Phase 2

Single-root MCTS with batched random rollouts

MCTS searches one root state at a time. Tree traversal is simple and debuggable; leaf evaluation is batched so it already resembles the future GPU workflow.

Selection score

score = -child.mean_value
      + c * sqrt(log(parent.visits + 1) / child.visits)

The minus sign is deliberate: child values are from the child player-to-move perspective, which is the opponent of the parent.

Backpropagation

for node in reversed(path):
    node.visits += 1
    node.value_sum += value
    value = -value

Every edge crosses to the other player, so the value flips sign at every step back to the root.

Objects and modules

How the pieces fit together

The package keeps rules, search, storage, and evaluation separate. That is why the later Zobrist DAG and neural-network evaluator can arrive as replacements instead of rewrites.

Connect4x4x4Batch

Owns batched board state and applies legal gravity moves.

  • board, heights, done, outcome
  • step(actions)
  • legal_mask()
  • check_wins()
  • clone()

MCTS

Coordinates selection, expansion, evaluation, and backprop.

  • search(root)
  • _select_and_expand()
  • _ucb_score()
  • _backpropagate()
  • _build_result()

NodeStore

Abstracts how search nodes are created and linked.

  • create_root(state)
  • get_or_create_child(...)
  • TreeNodeStore now
  • ZobristNodeStore later

RandomRolloutEvaluator

Replicates a leaf state and plays random games to termination.

  • evaluate(state)
  • batched random actions
  • CPU, MPS, or CUDA device
  • mean value in [-1, 1]

Value semantics

The sign convention that keeps search sane

This is the single most important mental model for AlphaZero-style code in this repo.

Engine step result

outcome = 1 means the player who just moved won.

It does not mean the next player-to-move is winning.

value flips

MCTS node value

A node value is always from that node's player-to-move perspective.

A winning terminal child is -1 for the child player.
